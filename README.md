# FP日報アプリ（React版）

Vite + React + Firebase構成。IN評価分析システムと同じ技術構成（Vercelデプロイ）です。

## 重要な仕様メモ

- **総販／リク抜き**：MNP・純新規・機種変更・アップの合計が総販。総販から機種変更とアップを除いたものがリク抜き。ダウンはどちらにもカウントしない（実績表示はするが集計対象外）。
- **キャッチャーのKPI**：総販・リク抜きの対象外。「着座件数」という独自の行動目標として、KPI画面の同じ表の中で直接入力する。
- **見込み獲得**：実績（au/UQなど）とは異なり、土曜・日曜どちらのタブを開いていても両方の入力欄が常に表示される。タブで隠れない。
- **重複保存判定**：西暦込みの日付・店舗名・ログイン中ユーザー名・メールアドレスの全項目が一致した場合のみ、上書き確認のモーダルを表示する。
- **LINE案件指示書の自動入力**：店舗名・日程・目標（総販）・日ごとの人員リストを抽出する。実績数値はこの種のメッセージに含まれないため対象外。人員リストは `fp_users`（ユーザー管理に登録済みの名前）と突き合わせて自社／他社を自動判定する。フォーマットは現場ごとに表記揺れがある前提で、ある程度緩く拾う設計にしている。実際の運用で誤判定が多い場合は調整が必要。

## セットアップ

```bash
npm install
npm run dev       # ローカル確認
npm run build     # 本番ビルド → dist/
```

## Vercelへのデプロイ

1. このフォルダ一式をGitHubリポジトリにpush（既存の `nippou` リポジトリに上書き）
2. Vercelは自動的に再デプロイ
3. `vercel.json` でSPAのルーティング（リロード時404）に対応済み

## Firebase設定

`src/lib/firebase.js` は既存の `nippou-data-base` プロジェクトのままなので、データはそのまま引き継がれます。匿名認証・Firebaseルールも変更不要です。

## GitHub Desktopでの反映手順（重要）

過去にフォルダ構造が崩れた事例があったため、必ず以下の手順で：

1. このzipを**展開（解凍）**する
2. GitHub Desktopでクローン済みのローカルフォルダを開く（「Show in Explorer」）
3. 中の古いファイルを**全部削除**する（`.git` フォルダは触らない）
4. 展開した `nippou-app` フォルダの**中身**（`src` フォルダそのものを含む）を、まとめて選択してコピー
5. 空にしたフォルダに貼り付け（`src` フォルダが1つの単位として存在することを確認）
6. GitHub Desktopで変更を確認 → Commit → Push

「`src` フォルダの中のファイルだけ」をコピーすると、フォルダ階層が崩れてビルドが失敗します。`src` フォルダごとコピーすること。
